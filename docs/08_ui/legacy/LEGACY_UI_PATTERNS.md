# LEGACY_UI_PATTERNS.md  
Legacy UI Patterns — UX Reference & Anti-Patterns

Статус: LEGACY / Reference-only  
Область: UI / UX / Historical patterns  
Назначение: фиксация UX-паттернов и анти-паттернов из legacy-прототипов и сторонних терминалов  
Ограничение: не является каноном и не даёт разрешений на реализацию

---

## 1. Назначение документа

Данный документ собирает **устойчивые UI-паттерны**, обнаруженные в:
- ранних собственных прототипах (Aegiron),
- сторонних торговых терминалах (MoonBot),
- экспериментальных UI-решениях до фиксации текущей архитектуры.

Цели:
- сохранить полезные UX-идеи;
- явно зафиксировать **опасные control-surface паттерны**;
- предотвратить возврат UI как источника власти;
- дать единый язык для обсуждения UI-решений.

Документ **не является инструкцией по реализации UI**.

---

## 2. Архитектурный контекст (точка отсчёта)

Все паттерны оцениваются **только** относительно канона MontrixBot:

- SIM = интеллект без власти  
- Gate = проверка права действовать (evidence-based)  
- REAL = исполнение без интеллекта  
- UI = наблюдение, объяснение, аудит (read-only)

Любой паттерн, нарушающий эту вертикаль, считается **архитектурно недопустимым**.

---

## 3. Классификация legacy UI-паттернов

### 3.1 Допустимые UX-паттерны (при изоляции власти)

Паттерны, которые можно использовать **визуально**, без переноса управления:

- плотные таблицы данных;
- фильтры и сортировки;
- accordion-группировка параметров;
- layout «список → detail panel»;
- отчётные формы;
- компактные summary-блоки.

### 3.2 Опасные паттерны (требуют переосмысления)

- экраны, выглядящие как “настройки”, но влияющие на исполнение;
- быстрые действия без explainability;
- автоматизация без Gate.

### 3.3 Запрещённые control-surface паттерны

Паттерны, которые **не могут появиться** в каноническом UI.

---

## 4. Запрещённые control-surface паттерны

### 4.1 Торговые кнопки в UI

Примеры:
- BUY / SELL
- Close position
- Cancel buy
- Market buy / market sell

**Почему запрещено:**  
UI получает прямую власть над REAL, минуя Gate и evidence.

---

### 4.2 Panic-действия из UI

Примеры:
- Panic Sell
- Emergency Close
- Kill trade button

**Почему запрещено:**  
Экстренные действия должны быть:
- частью политики,
- отражены в статусе,
- зафиксированы событиями,
а не кнопкой в интерфейсе.

---

### 4.3 Управление исполнением через “Settings”

Признаки:
- чекбоксы и слайдеры,
- которые фактически меняют вход/выход/объём.

**Почему запрещено:**  
Пользователь совершает действие, думая, что меняет настройку.

---

### 4.4 Стратегия как action

Примеры:
- Start / Stop strategy
- Enable strategy = начать торговать
- Strategy button с прямым эффектом

**Почему запрещено:**  
Стратегия должна быть **декларативным контрактом**, а не командой.

---

### 4.5 Auto-усиление без Gate

Примеры:
- auto-increase weights
- auto-enable strategies
- “best strategy now”

**Почему запрещено:**  
Скрытая передача власти алгоритму без проверяемого разрешения.

---

## 5. Паттерны, которые стоит перенять (UX-only)

### 5.1 Постоянная строка статуса

Идея:
- FSM
- Mode
- Gate decision
- HARD_STOP
- свежесть данных (age)

**Куда в MontrixBot:**  
Main + Explain.

---

### 5.2 WHY NOT как основной UX-объект

Идея:
- пользователь сначала видит причины блокировки,
- а не “что нажать”.

**Куда в MontrixBot:**  
Explain (первый экранный блок).

---

### 5.3 Таблицы + фильтры

Идея:
- быстрый обзор больших массивов данных;
- фильтрация по времени, статусу, confidence.

**Куда в MontrixBot:**  
Journal, Trade Viewer, возможный Market/Universe Viewer (read-only).

---

### 5.4 Layout «список → детали»

Идея:
- слева список сущностей,
- справа или снизу подробности выбранной.

**Куда в MontrixBot:**  
Strategies, Journal, Trade Viewer.

---

### 5.5 Быстрые переходы (не действия)

Из Aegiron:
- тайлы “Рост”, “Радар”, “Стратегии”.

**В MontrixBot:**  
только как:
- навигация,
- предустановленные фильтры,
- сценарии просмотра.

---

## 6. Переосмысление опасных паттернов

### 6.1 “Quick Buy” → Simulation / Preview

Возможная трансформация:
- не покупка,
- а симуляция или предпросмотр решения SIM.

---

### 6.2 Panic Sell → HARD_STOP + Explain

Эквивалент:
- политика безопасности,
- видимое состояние,
- журналируемое событие,
- объяснение причин.
